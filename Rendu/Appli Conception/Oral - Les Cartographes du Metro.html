0 · Vue d'ensemble & le jeu

« Les Cartographes du Métro » est un jeu de plateau (1 à 4 joueurs) où l'on relie des stations de métro à travers les arrondissements de Paris. Le logiciel que nous avons développé n'est pas le jeu complet : c'est l'éditeur de plateau (la partie « création et caractéristiques » décrite dans les règles). Il permet de :

  • Configurer une partie : largeur, hauteur, nombre de joueurs, nombre de types de stations, nombre d'arrondissements, taille des cases.
  • Dessiner le plateau : colorier chaque case avec un arrondissement (au pinceau, à la souris).
  • Placer les stations (types de métro) et les départs des joueurs.
  • Générer automatiquement le réseau : les arêtes (lignes de métro) qui relient les stations.
  • Sauvegarder / charger un plateau dans un fichier texte, avec contrôle d'intégrité.
L'idée clé à faire passer à l'oral Le plateau est un graphe : chaque station est un sommet, chaque ligne de métro est une arête. Le cœur « intelligent » du projet, c'est l'algorithme qui transforme un simple placement de stations en un graphe de connexions cohérent. Tout le reste (IHM, fichiers, validation) gravite autour de cette structure.

Les 3 « couleurs » de données stockées par case

DonnéeSignificationPlage de valeurs0 signifie
ArrondissementLa « couleur » de la case (zone de Paris)0 à 20Case vide (non coloriée)
StationType de métro posé sur la case (= un sommet du graphe)0 à 10 (6 utilisés)Pas de station
DépartBase de départ d'un joueur0 à 4Pas de départ

Une case peut cumuler les trois : par exemple « 18ᵉ arrondissement + station Moulin Rouge + départ du joueur 1 ».

1 · Architecture générale — le patron MVC

Le projet suit le patron MVC (Modèle – Vue – Contrôleur), matérialisé par trois paquets :

┌──────────────────────────────────────────────────────────────┐ │ plateau.ihm (la VUE) │ │ Frames + Panels Swing : Accueil, Configuration, Création, │ │ Jeu, CasePanel, PanelErreur, ListenerJeu │ └───────────────────────────────┬──────────────────────────────┘ │ appelle uniquement… ┌───────────────────────────────▼──────────────────────────────┐ │ plateau.Controleur (le CHEF D'ORCHESTRE) │ │ Façade : seul point de passage entre la Vue et le Modèle │ └───────────────────────────────┬──────────────────────────────┘ │ pilote… ┌───────────────────────────────▼──────────────────────────────┐ │ plateau.metier (le MODÈLE) │ │ Plateau, GraphePlateau, Chargeur, Enregistreur, Validateur, │ │ ConfigurationPlateau, GestionnairePlacement, UtilitaireJeu │ └──────────────────────────────────────────────────────────────┘
Pourquoi cette séparation ?
  • Découplage : la Vue ne connaît jamais la classe Plateau. Elle ne parle qu'au Controleur. On pourrait remplacer toute l'interface Swing (par une version web, console…) sans toucher au métier.
  • Responsabilité unique : le métier calcule, la vue affiche, le contrôleur fait le lien. Chaque bug se cherche au bon endroit.
  • Testabilité : la logique du graphe peut être testée sans lancer la moindre fenêtre.
Le sens de circulation (à savoir réciter) L'utilisateur clique sur une case → CasePanel prévient PanelJeu.caseCliquee() → qui appelle GestionnairePlacement.gererClic() via le Controleur → le Controleur modifie le Plateau → puis genererAretesAuto() recalcule le graphe → enfin repaint() redessine l'écran. La Vue ne décide rien, elle déclenche et affiche.

Bien distinguer extends et implements à l'oral

extends = héritage « EST-UN » FrameJeu extends JFrame : une FrameJeu EST une fenêtre. On récupère gratuitement tout le comportement d'une fenêtre (taille, fermeture, titre…) et on ne rajoute que notre spécificité. Java n'autorise qu'un seul extends (héritage simple).
implements = contrat « SAIT-FAIRE » implements ActionListener : la classe s'engage à savoir réagir à un clic (méthode actionPerformed). On peut implémenter plusieurs interfaces — c'est justement le cas de PanelCreation qui écoute à la fois la souris, le déplacement de souris et les boutons.
Pourquoi des classes « statiques » (utilitaires) au lieu d'objets ? Les classes ChargeurPlateau, EnregistreurPlateau, ValidateurPlateau, ConfigurationPlateau, GestionnairePlacement et UtilitaireJeu n'ont aucun attribut (aucun état à retenir). Ce sont des « boîtes à outils » : on leur donne des données, elles rendent un résultat. On les a donc faites statiques → pas besoin de créer un objet (ChargeurPlateau.charger(f) s'appelle directement). C'est l'application du principe de responsabilité unique : on a sorti ces traitements de la classe Plateau pour qu'elle ne fasse que stocker des données.
Question piège fréquente « Pourquoi avoir séparé GraphePlateau de Plateau ? » → Parce que ce sont deux responsabilités différentes : Plateau gère le contenu des cases (couleurs, stations, départs), GraphePlateau gère les connexions. Si demain on change la façon de stocker les arêtes (liste au lieu d'une matrice), Plateau n'est pas touché.

3 · Côté MÉTIER — méthode par méthode

Pour chaque méthode importante : ⚙️ comment ça marche, 🎯 pourquoi ce choix, 🔧 où / comment on l'utilise.

Plateau — le modèle de données

C'est le « plan » du plateau. Il ne calcule rien : il retient. Il contient trois tableaux d'entiers et un graphe.

private int largeur, hauteur;
private int[] tabArrondissements;   // la couleur de chaque case
private int[] tabStations;          // le type de station de chaque case
private int[] tabDeparts;           // le départ joueur de chaque case
private GraphePlateau graphe;       // les connexions entre stations
Choix n°1 : trois tableaux à UNE dimension (et pas un tableau 2D ni des objets Case) Le plateau est une grille 2D, mais on la stocke « à plat » dans des tableaux 1D. Une case (x, y) est rangée à l'indice i = y × largeur + x. Inversement : x = i % largeur et y = i / largeur. Pourquoi ? (1) Un seul indice suffit pour désigner une case, ce qui simplifie énormément l'IHM (chaque CasePanel ne retient qu'un int index) ; (2) le format de fichier devient trivial (une ligne = un indice) ; (3) c'est exactement la même indexation que la matrice du graphe. Trois tableaux parallèles plutôt qu'une classe Case : plus léger et direct pour un plateau « tableur ». alternative : une classe Case avec 3 attributs

public Plateau(int largeur, int hauteur)

Comment Calcule taille = largeur × hauteur, crée les trois tableaux de cette taille (Java les initialise à 0 = « vide ») et instancie un GraphePlateau(taille).
Pourquoi Tout est dimensionné une seule fois, à la construction. Le 0 par défaut de Java nous arrange : une case neuve est automatiquement « sans arrondissement, sans station, sans départ ».
Utilisation Appelé par Controleur.initialiserPlateau() (nouvelle partie) et par ChargeurPlateau (chargement).

get/affecter Arrondissement / Station / Depart (numCase, valeur)

Comment Chaque accesseur teste d'abord que numCase est dans les bornes du tableau avant de lire/écrire. En lecture hors-bornes, on renvoie 0 (« vide ») ; en écriture hors-bornes, on ne fait rien.
Pourquoi cette « programmation défensive » Les indices viennent de clics souris et de fichiers externes : ils peuvent être faux. En blindant l'accès, on évite un ArrayIndexOutOfBoundsException qui ferait planter le jeu. C'est un choix de robustesse.
Utilisation Ce sont les seules portes d'entrée vers les données ; tout passe par elles (IHM via Controleur, Chargeur, Enregistreur).

GraphePlateau — le graphe (matrice d'adjacence)

private boolean[][] matriceAretes;   // matriceAretes[i][j] == true  ⟺  arête entre i et j
Choix n°2 : une matrice d'adjacence boolean[taille][taille] Un graphe peut se représenter de deux façons : une matrice d'adjacence (un grand tableau 2D de booléens) ou des listes d'adjacence (chaque sommet garde la liste de ses voisins). Nous avons choisi la matrice car :
  • Tester « y a-t-il une arête entre i et j ? » est immédiat (O(1)) : une simple lecture matriceAretes[i][j].
  • Le graphe est non orienté : on garantit la symétrie en écrivant [i][j] ET [j][i] en même temps.
  • C'est simple à comprendre et à coder ; les plateaux visés sont petits (7×7, 10×10), donc le coût mémoire est négligeable.
limite assumée : pour un plateau géant, une liste d'adjacence serait plus économe (voir §10)

public void ajouterArete(int i, int j)

Comment Après vérification des bornes, met matriceAretes[i][j] = true et matriceAretes[j][i] = true.
Pourquoi écrire les deux Une ligne de métro n'a pas de sens : « i relié à j » = « j relié à i ». Écrire les deux cases rend la matrice symétrique, donc aArete(i,j) et aArete(j,i) donnent toujours la même réponse.
Utilisation Appelée par genererAretesAuto() et par ChargeurPlateau (relecture des arêtes d'un fichier).

public int getNbAretes()

Comment Double boucle, mais la boucle interne démarre à j = i + 1 (triangle supérieur de la matrice) et compte les true.
Pourquoi j = i + 1 et pas j = 0 La matrice étant symétrique, l'arête {i, j} apparaît deux fois (en [i][j] et [j][i]). En ne parcourant que la moitié au-dessus de la diagonale, on compte chaque arête une seule fois. Sinon on afficherait le double.
Utilisation Affiché à l'écran : « Nombre d'arêtes : … ». C'est une donnée de contrôle pour l'utilisateur.

public void vider() / aDesAretes()

Comment vider() repasse toute la matrice à false ; aDesAretes() renvoie true dès qu'elle trouve une arête.
Pourquoi vider() garantit que la régénération du graphe repart de zéro (idempotence : recalculer deux fois donne le même résultat, sans arêtes fantômes).
Utilisation vider() est la première ligne de genererAretesAuto().

genererAretesAuto() est le cœur du projet — il a sa propre section détaillée : voir §4 · Sélection des arêtes.

ChargeurPlateau — lecture & validation d'un fichier

public static Plateau charger(File fichier)

Comment Lit le fichier ligne par ligne avec un Scanner. La 1ʳᵉ ligne donne largeur;hauteur. Les taille lignes suivantes décrivent les cases (indice;arrondissement;station;depart). Les lignes restantes décrivent les arêtes (source;destination). À chaque étape, une série de contrôles peut interrompre la lecture en renvoyant null.
Pourquoi renvoyer null et pas planter Un fichier peut être corrompu, modifié à la main, incomplet. Plutôt que de laisser une exception remonter, on renvoie null = « chargement échoué ». L'IHM le détecte et affiche proprement le PanelErreur. On ne fait jamais confiance à une donnée externe.
Utilisation Appelée par Controleur.chargerPlateau() quand on clique « Charger le plateau ».

Le détail des 8 contrôles est en §5 · Vérifications & contraintes.

EnregistreurPlateau — écriture d'un fichier

public static boolean enregistrer(Plateau plateau, String nomFichier)

Comment (1) Régénère les arêtes (genererAretesAuto) pour sauver un graphe à jour ; (2) crée le dossier sauvegarde/ s'il n'existe pas ; (3) ajoute l'extension .txt si l'utilisateur l'a oubliée ; (4) écrit l'en-tête, puis une ligne par case, puis une ligne par arête (uniquement le triangle supérieur j > i).
Pourquoi régénérer les arêtes au moment de sauver Pour garantir que le fichier reflète exactement le placement actuel des stations, même si l'utilisateur a déplacé des stations juste avant. Pourquoi écrire j > i : même raison que getNbAretes() — on n'écrit chaque arête qu'une fois pour ne pas doubler le fichier.
Utilisation Bouton « Valider » de la création et « Sauvegarder les modifications » de l'écran de jeu.

ValidateurPlateau — vérifier une règle du jeu

public static boolean validerDeparts(Plateau plateau, int nbJoueurs)

Comment Pour chaque joueur de 1 à nbJoueurs, parcourt toutes les cases et compte combien portent son départ. Si ce n'est pas exactement 1, renvoie false (et explique l'erreur en console).
Pourquoi exactement 1 C'est une règle du jeu : « chaque joueur place SA station de départ » (1 joueur = 1 ligne = 1 base). Zéro départ = joueur sans réseau ; deux départs = triche. La validation empêche de sauvegarder un plateau injouable.
Utilisation Appelée avant la sauvegarde finale dans ListenerJeu (bouton « Sauvegarder »).

ConfigurationPlateau — déduire la config d'un plateau chargé

public static int[] detecterMaxJoueursEtStations(Plateau, int nbJoueursDefaut, int nbStationsDefaut)

Comment Parcourt toutes les cases et garde le maximum des numéros de station et des numéros de départ rencontrés. Renvoie un tableau {maxJoueurs, maxStations}.
Pourquoi Quand on charge un plateau existant, on ne connaît pas sa configuration d'origine. Plutôt que de la demander, on la déduit du contenu : s'il existe un « Joueur 3 » quelque part, c'est qu'il y a au moins 3 joueurs. L'interface s'adapte alors automatiquement (les menus déroulants proposent le bon nombre de joueurs/stations).
Utilisation Controleur.mettreAJourConfigurationDepuisPlateau(), juste après un chargement réussi.

GestionnairePlacement — la logique de pose

public static void gererClic(int index, String modeSelection, Object selectionItem, Controleur ctrl)

Comment Selon le mode (« STATION » ou « DEPART ») et l'élément choisi dans le menu déroulant : convertit le libellé (« Tour Eiffel », « Joueur 2 », « Aucun ») en numéro, puis appelle ctrl.affecterStation() ou ctrl.affecterDepart(). Cas spécial des départs : avant de poser le départ du joueur N, il efface l'ancien départ de ce même joueur ailleurs sur le plateau.
Pourquoi effacer l'ancien départ Pour garantir la règle « 1 seul départ par joueur » dès la saisie, et non pas seulement à la validation. C'est de la prévention : l'utilisateur ne peut pas créer deux départs pour le même joueur. Pourquoi sortir cette logique de l'IHM : transformer un libellé texte en numéro est de la logique « métier », elle n'a pas sa place dans le code d'affichage.
Utilisation Appelée par PanelJeu.caseCliquee() à chaque clic sur une case en mode édition.

UtilitaireJeu — les constantes partagées

getNomsStations() · getCouleurs() · getCheminImageStation(int)

Comment Renvoient respectivement le tableau des noms de stations (« Tour Eiffel », « Louvre »…), le tableau des 20 couleurs d'arrondissement, et le chemin d'une image de station. getCheminImageStation essaie plusieurs chemins (plateau/images/, ../plateau/images/…) et renvoie le premier qui existe.
Pourquoi centraliser ces constantes Les couleurs et noms sont utilisés à plusieurs endroits (création ET jeu). En les définissant à un seul endroit, on évite les incohérences (même couleur pour le même arrondissement partout) et on n'a qu'un seul fichier à modifier. Pourquoi tester plusieurs chemins d'image : le programme peut être lancé depuis différents dossiers de travail ; on rend le chargement des images robuste quel que soit le répertoire courant.
Utilisation Couleurs → CasePanel et PanelCreation ; noms → menus déroulants ; chemins → cache d'images de PanelJeu.

Controleur — la façade

Le contrôleur est presque entièrement composé de méthodes « passe-plat » qui délèguent au métier (getLargeur() → metier.getLargeur()). Quelques méthodes orchestrent plusieurs classes métier :

C'est exactement le rôle d'une façade.

⭐ 4 · La sélection des arêtes (le sujet n°1)

C'est l'algorithme à maîtriser parfaitement. Il transforme un placement de stations en un réseau de lignes de métro.

public void genererAretesAuto(Plateau plateau) {
    int taille = plateau.getLargeur() * plateau.getHauteur();
    this.vider();                                  // on repart de zéro

    // Les 4 directions « vers le bas / la droite » : Est, Sud-Est, Sud, Sud-Ouest
    int[] deplacementsX = {1, 1, 0, -1};
    int[] deplacementsY = {0, 1, 1,  1};

    for (int i = 0; i < taille; i++) {
        if (plateau.getStation(i) == 0) continue;  // pas de station = pas un sommet

        int x = i % plateau.getLargeur();          // indice 1D → coordonnées 2D
        int y = i / plateau.getLargeur();

        for (int direction = 0; direction < 4; direction++) {
            int cx = x + deplacementsX[direction];
            int cy = y + deplacementsY[direction];

            // on avance dans la direction tant qu'on reste sur le plateau…
            while (cx >= 0 && cx < plateau.getLargeur() && cy >= 0 && cy < plateau.getHauteur()) {
                int indexCible = cy * plateau.getLargeur() + cx;
                if (plateau.getStation(indexCible) > 0) {  // première station trouvée ?
                    this.ajouterArete(i, indexCible);
                    break;                              // on s'arrête au sommet le plus proche
                }
                cx += deplacementsX[direction];          // sinon on continue tout droit
                cy += deplacementsY[direction];
            }
        }
    }
}

Comment ça marche, en 4 idées

1. On ne relie QUE les stations On saute toute case dont getStation(i) == 0. Les sommets du graphe sont les stations, pas les cases.
2. Conversion indice ⇄ coordonnées Comme tout est stocké à plat, on retrouve la colonne et la ligne d'une station avec x = i % largeur et y = i / largeur, et on revient à l'indice avec indice = y × largeur + x. C'est ce qui permet de « se déplacer » dans la grille.
3. Un « rayon » jusqu'à la station la plus proche Dans chaque direction, la boucle while avance case par case. Dès qu'elle rencontre une station, elle pose l'arête et fait break. On traverse les cases vides (sans station) et on s'arrête à la première station : c'est « la prochaine station sur la ligne », même si elle est loin.
4. On repart de zéro à chaque appel vider() au début → l'algorithme est rejouable : on peut le relancer après chaque clic sans accumuler de vieilles arêtes.

🎯 LE choix à justifier : pourquoi 4 directions et pas 8 ?

Le raisonnement complet (à connaître par cœur) Une station peut se connecter dans 8 directions (E, SE, S, SO, O, NO, N, NE — comme les règles du jeu). Mais le graphe est non orienté : l'arête entre A (au nord) et B (au sud) est la même, qu'on la regarde « du nord vers le sud » ou « du sud vers le nord ».

On a donc choisi 4 directions seulement : les 4 qui pointent « vers la droite / vers le bas » (Est, Sud-Est, Sud, Sud-Ouest). Comme ajouterArete écrit la liaison dans les deux sens, quand on traite la station du nord, on crée déjà la liaison avec celle du sud. Quand viendra le tour de la station du sud, sa direction « Nord » n'a plus besoin d'être explorée : l'arête existe déjà.

Résultat : les 4 directions « avant » de chaque sommet couvrent automatiquement les 4 directions « arrière » des autres sommets. On obtient bien les 8 connexions possibles, mais avec deux fois moins de travail et aucune arête en double.
NO N NE Directions EXPLORÉES (4) : Est, Sud-Est, Sud, Sud-Ouest \ | / Directions DÉDUITES (4) : Ouest, Nord-Ouest, Nord, Nord-Est \ | / (obtenues gratuitement par la symétrie de la matrice) O ───────[ STATION ]─────── E / | \ / | \ SO S SE orange = exploré directement · vert = déduit par une autre station
Pourquoi la génération AUTOMATIQUE plutôt que tracer les lignes à la main ? Le concepteur ne place que les stations ; le logiciel en déduit tout le réseau. Avantages : (1) pas d'erreur humaine (on ne peut pas oublier ou doubler une ligne) ; (2) cohérence garantie (la même règle s'applique partout) ; (3) mise à jour instantanée : on déplace une station, le réseau se recalcule tout seul. C'est ce qui rend l'éditeur agréable et fiable.
Questions probables sur cet algorithme
  • « Et si vous mettiez 8 directions ? » → Ça marcherait aussi, mais on créerait chaque arête deux fois (travail inutile) ; la symétrie de la matrice nous l'évite.
  • « Pourquoi le break ? » → Pour ne relier qu'à la station la plus proche dans la ligne, pas à toutes les stations alignées derrière.
  • « Pourquoi continue si station == 0 ? » → Une case sans station n'est pas un sommet du graphe ; inutile d'explorer depuis elle.
  • « Complexité ? » → Pour chaque case (N = largeur×hauteur), on explore 4 directions sur une longueur max ~max(largeur,hauteur). Soit de l'ordre de O(N × côté). Sur des plateaux 7×7 à 10×10, c'est instantané.

5 · Algorithmes de vérification & contraintes

Le programme refuse les données incohérentes à trois moments : à la saisie (configuration), à la validation du plateau (création / jeu) et au chargement d'un fichier. C'est la « défense en profondeur ».

5.1 · La « coloration » : contraintes sur les arrondissements

Point important à clarifier devant le jury Dans un problème classique de coloration de graphe, la contrainte est : « deux sommets adjacents doivent avoir des couleurs différentes ». Ce n'est PAS le cas ici, et c'est volontaire : un arrondissement de Paris est une zone qui s'étale sur plusieurs cases contiguës de même couleur. Deux cases voisines peuvent et doivent pouvoir partager la même couleur. Ici, « coloration » = affecter un arrondissement (une couleur) à chaque case.

Les contraintes réellement vérifiées sur la coloration sont :

ContrainteOùComment c'est vérifié
Toutes les cases doivent être coloriées avant de validerPanelCreation (bouton Valider)On parcourt tabCases : si une case est encore LIGHT_GRAY (gris = vide), on bloque la validation
Nombre d'arrondissements ≤ 20Config + Chargeur20 = le nombre réel d'arrondissements de Paris ; valeur bornée à la saisie et au chargement (arr 0..20)
Une seule couleur par caseModèleGaranti par construction : un seul int par case dans tabArrondissements

5.2 · Les 8 contrôles du chargement de fichier

La méthode ChargeurPlateau.charger() applique une vérification à chaque étape. Si une seule échoue → null → écran d'erreur.

#ContrôlePourquoi
1Le fichier existe et est lisibleÉviter une exception d'accès disque
2L'en-tête a exactement 2 valeursSans largeur ET hauteur, on ne sait rien dimensionner
3Dimensions positives et < 100Refuser des tailles absurdes (négatives, ou un plateau de 10 000×10 000)
4Une ligne de case a au moins 2 champsIl faut au minimum un indice et un arrondissement
5Les indices se suivent de 0 à taille−1Détecter une ligne manquante ou en désordre
6Valeurs dans les bornes (arr ≤ 20, station ≤ 10, départ ≤ 4)Refuser un « arrondissement 57 » impossible
7Les extrémités d'une arête existent sur le plateauPas d'arête vers une case inexistante
8On a lu EXACTEMENT taille casesDétecter un fichier tronqué
+ Filets de sécurité Le tout est dans un try/catch : un NumberFormatException (quelqu'un a écrit « A;B » au lieu de chiffres) est attrapé et transformé en « fichier corrompu ». Et si le fichier ne contenait aucune arête, on les génère automatiquement (genererAretesAuto) — pratique pour importer un plateau décrit uniquement par ses stations.

5.3 · Contrôles de saisie (formulaire de configuration)

Dans PanelConfiguration, avant de créer le plateau : champs non vides, valeurs numériques (try/catch sur Integer.parseInt), bornes maximales (joueurs ≤ 4, stations ≤ 6, arrondissements ≤ 20), valeurs strictement positives, et cohérence (la largeur et la hauteur doivent être ≥ au nombre de joueurs, sinon on ne pourrait pas placer tous les départs).

5.4 · Contrôle de la règle « 1 départ par joueur »

Géré deux fois : en prévention (GestionnairePlacement efface l'ancien départ avant d'en poser un nouveau) et en validation finale (ValidateurPlateau.validerDeparts exige exactement 1 départ par joueur avant la sauvegarde).

Pourquoi vérifier au même endroit plusieurs fois (à la saisie ET au chargement) ? Parce que les données peuvent arriver par deux chemins : l'interface (qu'on maîtrise) et un fichier (qu'un utilisateur peut éditer à la main). On ne peut faire confiance qu'à ce qu'on a soi-même contrôlé : on valide donc à chaque frontière du programme.

6 · Côté IHM (Swing) — le code et ses justifications

6.1 · La hiérarchie Frame / Panel

Chaque écran = une Frame (la fenêtre, extends JFrame) qui contient un Panel (le contenu, extends JPanel). La Frame est volontairement minimaliste : elle fixe le titre, la taille, le centrage et délègue tout l'affichage au Panel.

Pourquoi séparer Frame et Panel ? Un JPanel est réutilisable et imbriquable (on peut mettre un panel dans un panel), alors qu'une JFrame est une fenêtre du système. En mettant le contenu dans un Panel, on garde la possibilité de le réorganiser, et la Frame ne porte aucune logique.

6.2 · Les gestionnaires de disposition (Layouts) — chaque choix a une raison

LayoutOùPourquoi celui-là

GridLayout
(grille de cases identiques)		Le plateau, le formulaire, les boutons d'arrondissementToutes les cellules ont exactement la même taille → parfait pour une grille de jeu régulière (hauteur×largeur cases) et pour un formulaire « libellé / champ ».

6.3 · Les « Box » — la question que tu m'as posée

this.panelGauche.setLayout(new BoxLayout(this.panelGauche, BoxLayout.Y_AXIS));
...
this.panelGauche.add(this.lblJoueurs);
this.panelGauche.add(Box.createVerticalStrut(10));   // ← espace vide de 10 px
this.panelGauche.add(this.lblStations);
this.panelGauche.add(Box.createVerticalStrut(10));
Ce que c'est BoxLayout.Y_AXIS empile les composants les uns sous les autres (colonne). Box.createVerticalStrut(10) insère un séparateur invisible de 10 pixels de haut entre deux composants. C'est un « espaceur ».
Pourquoi utiliser des Box plutôt qu'autre chose ?
  • Un GridLayout imposerait à tous les éléments la même hauteur : impossible d'aérer sélectivement (un petit espace ici, un grand là).
  • Avec les strut, on contrôle l'espacement au pixel près, élément par élément (10 px entre les infos, 20 px avant une nouvelle section, 30 px avant le bouton sauvegarder). C'est ce qui donne une colonne lisible et hiérarchisée.
  • C'est plus simple que de calculer des marges avec un GridBagLayout pour un simple empilement vertical.
Les compagnons des Box setAlignmentX(LEFT_ALIGNMENT) force tous les composants à s'aligner à gauche (sinon BoxLayout les centre). setMaximumSize(...) empêche un menu déroulant de s'étirer sur toute la hauteur disponible. setBorder(createTitledBorder("Informations du Jeu")) dessine un cadre titré autour du bloc.

6.4 · Le dessin personnalisé : paintComponent vs paint

a) CasePanel : dessiner une case (paintComponent)

protected void paintComponent(Graphics g) {
    super.paintComponent(g);
    // 1. fond = couleur de l'arrondissement
    int arr = ctrl.getArrondissement(index);
    if (arr == 0) g.setColor(Color.LIGHT_GRAY);
    else g.setColor(tabCouleurs[arr - 1]);
    g.fillRect(0, 0, w, h);
    // 2. image de la station par-dessus
    if (station > 0) g.drawImage(panelJeu.getImageStation(station), ...);
    // 3. texte "D1", "D2"… si départ
    if (depart > 0) g.drawString("D" + depart, 5, 18);
}
Pourquoi redéfinir paintComponent ? C'est la méthode recommandée par Swing pour dessiner le contenu d'un composant (elle profite du double buffering, donc pas de clignotement). On y superpose trois couches dans l'ordre : fond coloré → image de station → texte du départ. Chaque case se dessine elle-même à partir des données lues via le contrôleur : la Vue ne stocke rien, elle reflète le Modèle.

b) panelApercuGrille : dessiner les lignes de métro (paint)

this.panelApercuGrille = new JPanel() {
    public void paint(Graphics g) {
        super.paint(g);                  // dessine d'ABORD les cases enfants
        Graphics2D arete = (Graphics2D) g;
        arete.setRenderingHint(KEY_ANTIALIASING, VALUE_ANTIALIAS_ON);  // lignes lissées
        arete.setStroke(new BasicStroke(3));   // épaisseur 3 px
        // pour chaque arête, tracer une ligne entre les centres des 2 cases
        for (...) if (ctrl.aArete(i, j)) arete.drawLine(x1, y1, x2, y2);
    }
};
Pourquoi paint ici et pas paintComponent ? Les lignes de métro doivent apparaître PAR-DESSUS les cases. Or les composants enfants (les CasePanel) sont dessinés après le paintComponent du parent. En redéfinissant paint et en appelant super.paint(g) en premier (qui dessine les enfants), puis nos lignes, on est sûr que les arêtes se voient au-dessus de la grille. C'est un choix précis lié à l'ordre de rendu de Swing.
Pourquoi Graphics2D ? Graphics2D (sous-classe de Graphics) offre l'anti-aliasing (lignes lissées, pas en escalier) et le réglage de l'épaisseur du trait (BasicStroke). Une ligne de métro grise de 3 px, lissée, est bien plus lisible qu'un trait fin pixelisé.

6.6 · Le cache d'images des stations

private Image[] stationImages = new Image[11];

public Image getImageStation(int n) {
    if (stationImages[n] == null)                 // pas encore chargée ?
        stationImages[n] = new ImageIcon(chemin).getImage();  // on la charge UNE fois
    return stationImages[n];                       // puis on la réutilise
}
Pourquoi un cache (mémoïsation) ? Chaque case se redessine très souvent (à chaque repaint). Sans cache, on rechargerait le fichier image du disque à chaque dessin → lent et saccadé. On charge donc chaque image une seule fois, on la garde en mémoire, et on la réutilise. C'est un choix de performance classique.

6.7 · L'écran d'erreur dédié (PanelErreur)

Pourquoi une classe entière juste pour un message d'erreur ? Quand un fichier est corrompu, plutôt que d'afficher un message brut dans la console (invisible pour l'utilisateur), on affiche un panneau clair et rouge à la place du plateau. En faire une classe (PanelErreur) rend le code de PanelJeu plus lisible et le message réutilisable. C'est un souci d'expérience utilisateur.

9 · Questions probables du jury (avec réponses)

« Expliquez votre algorithme de génération des arêtes. » On parcourt chaque station ; pour chacune on regarde dans 4 des 8 directions (les 4 « vers l'avant »), on avance jusqu'à trouver la station la plus proche, et on crée une arête. Les 4 autres directions sont couvertes gratuitement car la matrice est symétrique. (→ détailler avec le schéma du §4.)
« Pourquoi une matrice et pas une liste d'adjacence ? » Test d'arête en temps constant, symétrie triviale à maintenir, code simple ; et nos plateaux sont petits donc le coût mémoire d'une matrice est négligeable. Pour un très grand plateau, une liste serait plus économe — c'est une amélioration identifiée.
« C'est de la coloration de graphe ? Les cases voisines doivent être de couleurs différentes ? » Non, et c'est volontaire. Ici la couleur = l'arrondissement, et un arrondissement est une zone faite de plusieurs cases voisines de même couleur. La contrainte qu'on vérifie est différente : toutes les cases doivent être coloriées avant de valider, et le nombre d'arrondissements est borné à 20.
« Où est le MVC ? Donnez le chemin d'un clic. » Clic sur une case → CasePanel → PanelJeu.caseCliquee → GestionnairePlacement.gererClic via le Controleur → modification du Plateau → genererAretesAuto → repaint. La Vue déclenche, le Métier calcule, le Contrôleur relie.
« Que se passe-t-il si on charge un fichier corrompu ? » ChargeurPlateau applique 8 contrôles + un try/catch. Au moindre problème, il renvoie null, et l'IHM affiche le PanelErreur sans planter.
« Pourquoi des méthodes statiques ? » Ces classes n'ont pas d'état à mémoriser : ce sont des fonctions pures (entrée → sortie). Le static évite de créer des objets inutiles et exprime bien qu'il s'agit d'outils.
« Comment garantissez-vous qu'un joueur n'a qu'un seul départ ? » Deux niveaux : à la pose, on efface l'ancien départ du joueur (GestionnairePlacement) ; à la sauvegarde, ValidateurPlateau exige exactement 1 départ par joueur.
« Pourquoi j = i+1 quand vous comptez/écrivez les arêtes ? » La matrice est symétrique : l'arête {i,j} y figure deux fois. On ne parcourt que le triangle supérieur pour la compter / l'écrire une seule fois.
« À quoi sert le contrôleur, à part recopier des appels ? » C'est la façade : il définit ce que la Vue a le droit de faire et isole le métier. Si le modèle change, seul le contrôleur est touché, pas les 11 classes d'interface.

10 · Limites assumées & améliorations possibles

Montrer qu'on a du recul sur son code est très apprécié à l'oral.

  • Matrice d'adjacence et grands plateaux : la mémoire croît en taille². Pour un plateau immense, une liste d'adjacence serait plus adaptée.
  • Champs statiques publics dans PanelConfiguration (txtLargeur…) : pratiques mais c'est un raccourci ; il serait plus propre de passer les valeurs par le contrôleur.
  • Messages d'erreur en console (System.out.println) : un utilisateur ne les voit pas ; on pourrait tout faire passer par des JOptionPane / panneaux visuels.
  • Pas d'objet Case : trois tableaux parallèles fonctionnent mais une classe Case serait plus orientée objet et plus extensible.
  • Le jeu lui-même n'est pas implémenté : on a l'éditeur de plateau et le calcul du graphe ; le déroulement des manches et le calcul des scores (décrits dans les règles) restent à faire.
